## 핵심 내용

- Claude 생태계의 경쟁 축이 "누가 더 똑똑한가"에서 "누가 실제 파일, 테스트, Git, 브라우저, 외부 도구, 장기 세션을 더 잘 묶는가"로 이동했다. (p.7)
- 차이는 모델 이름보다 작업 구조, 하네스, 검증 습관에서 더 많이 벌어진다. (p.3)
- Claude는 4개 작업 표면으로 구분된다. Chat(상담 창구), Projects(배경 자료 캐비닛), Cowork(실무 책상), Claude Code(개발 작업대). 같은 모델을 써도 어디에 앉히느냐에 따라 잘하는 일이 달라진다. (p.9–p.10)
- 하네스는 "어떤 파일을 읽게 할지, 어디까지 수정할 수 있는지, 무엇을 확인한 뒤 끝낼지, 결과를 어디에 남길지를 묶어 둔 바깥 구조"다. 모델이 강해질수록 '너무 넓게 고침', '검증 없이 끝냄'이 더 큰 문제가 된다. (p.8)
- Cowork와 Claude Code의 구분 기준은 난이도가 아니라 위임 중심인지(Cowork), 통제 중심인지(Claude Code)다. (p.27–p.28)
- 6개월 이상 사용한 고숙련 사용자일수록 더 높은 가치의 과업을 더 자주 다루고 대화 성공률도 높다. (Anthropic Economic Index, p.7, p.11)
- Claude의 확장 큰 흐름: "질문 도구 → 작업 도구 → 팀 운영 도구" (p.11)

## 내가 이해한 점

Claude를 쓰면서 단순히 답변을 잘 받는 법이 아니라, 클로드를 실제 작업에 어떻게 안정적으로 연결해야 하는지 더 자세히 알고 싶었다. 이 자료는 그 질문에 대한 출발점이 됐다.

이 자료의 핵심은 Claude를 단순한 대화형 AI가 아니라 실제 작업을 수행하는 시스템으로 봐야 한다는 점이다. 좋은 결과는 모델 성능만으로 나오는 것이 아니라 작업 구조, 하네스, 검증 습관에서 나온다는 것을 이해했다. 특히 하네스 개념이 중요했다. AI에게 막연히 요청하는 것이 아니라, 어떤 파일을 읽을지, 어디까지 수정할 수 있는지, 무엇을 확인하고 끝낼지, 결과를 어디에 남길지를 미리 정해줘야 한다. AI를 잘 쓰는 것은 프롬프트를 잘 쓰는 것뿐만 아니라, AI가 일할 수 있는 작업 환경과 기준을 설계하는 일에 가깝다는 것을 느꼈다.

또 Cowork와 Claude Code의 차이가 단순히 쉬운 도구와 어려운 도구의 차이가 아니라, 위임 중심인지 통제 중심인지의 차이라는 점이 기억에 남았다.

## 내 학습이나 프로젝트에 연결할 점

지금 만들고 있는 AI Learning Note Skill Kit과 직접 연결된다. 이 워크플로우에서도 AI가 정리할 영역(Learning Summary)과 내가 직접 써야 할 영역(My Note)을 분리하는 구조가 이 자료에서 말하는 하네스 개념과 비슷하다. 입력 파일, 출력 파일, 작성 범위, 금지 사항, 검토 기준을 미리 정해두는 방식이기 때문이다.

앞으로 내 워크플로우를 더 안정적으로 만들려면 Claude Code를 Writer 역할로, Codex를 Reviewer 역할로 나누는 구조를 더 명확히 해야 한다. Claude Code는 자료를 읽고 Summary·My Note Guide·README Draft를 만드는 쪽에 두고, Codex는 AI가 내 해석을 지어내지 않았는지, 자료에 없는 내용을 단정하지 않았는지 검토하는 역할로 둘 수 있다. 또 `CLAUDE.md`, `plan.md`, `work-log`, `handoff` 같은 파일을 활용하면 AI가 작업 맥락을 잃지 않고 이어갈 수 있을 것 같다.
