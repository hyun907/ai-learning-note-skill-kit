## p.3

왜 지금 이 책을 여는가
지금은 AI를 "무엇을 물어볼까"의 관점으로만 보기 어려운 시기다. 지난 1~2년 동안 바이
브 코딩(vibe coding) 이라는 말이 퍼졌고, 그 흐름은 문서 초안, 조사, 보고, 검토, 자동
화, 배포처럼 더 넓은 작업으로 번졌다. 중요한 질문도 "어느 모델이 더 똑똑한가"보다 "어
느 도구가 실제 일을 덜 끊기게 하는가"로 이동했다. 이 흐름은 안드레이 카파시(Andrej
Karpathy)가 말한 소프트웨어 3.0(Software 3.0) , 최근 업계가 말하는 LLM OS(거대
언어모델 작업 운영층) 와도 이어진다.
연구와 실무 보고서도 같은 방향을 가리킨다. METR은 에이전트가 감당하는 작업 길이가
빠르게 늘고 있다고 보고하고, Anthropic은 짧은 답변보다 조금 더 긴 작업을 맡기고 검증
하는 구조에서 체감이 커진다고 설명한다. 차이는 모델 이름보다 작업 구조, 하네스, 검증
습관에서 더 많이 벌어진다. 이 책은 AI를 돈버는 기계보다 가치 창출을 증폭하는 층으로
본다.
왜 이 책은 Claude Code를 중심에 두는가
Claude Code를 중심에 두는 이유는 분명하다. Anthropic이 어디로 가고 있는지는 Claude
Code를 보면 가장 선명하게 읽힌다. 파일, 규칙, 검증, 권한, 반복 작업, 원격 이어받기, 외부
도구 연결, 팀 배포 같은 문제가 한꺼번에 모여 있기 때문이다.
Cowork는 이 흐름을 비개발 업무로 확장한다. Anthropic 공식 릴리스 노트도 Cowork를
"Claude Code의 에이전트 기능을 코딩 바깥 지식 업무로 가져온 표면"으로 설명한다. 그래
서 Claude Code를 중심축으로 보면 Cowork와 Skills, Plugins, Connectors, Remote
Control, Scheduled Tasks 같은 요소도 자연스럽게 연결된다.
아래 그림은 이 변화를 압축한 지도다. 세부 기능보다 먼저, 질문창 경쟁이 왜 작업공간 경
쟁으로 옮겨 가는지만 잡으면 충분하다.

## p.4

이 책이 다루는 일도 결국 몇 가지 장면으로 모인다. 어떤 장면에서는 자료를 읽혀 주간 브
리프를 만든다. 어떤 장면에서는 회의 메모를 문서로 다시 정리한다. 또 어떤 장면에서는 작
은 버그를 고치고 테스트를 돌리고, 다음 사람이 이어받을 메모를 남긴다. 겉으로는 서로 다
른 일처럼 보여도, 실제로는 읽을 자료, 작업 범위, 검토 지점, 저장 위치를 어떻게 잡느냐라
는 같은 문제 위에 놓여 있다.
첫 번째 장면은 주간 브리프다. 경쟁사 메모와 기사 링크, 지난주 액션 아이템이 한 폴더에
있고 결과 형식이 템플릿으로 고정되어 있으면 Claude는 이를 이번 주 변화, 우리에게 의
미 있는 변화, 이번 주 액션 3개로 압축할 수 있다.
두 번째 장면은 회의 이후 문서화다. 회의 메모, 문서 양식, 아직 답이 안 난 질문 목록이 분
리돼 있으면 Claude는 회의 내용을 "무슨 말이 오갔는가"가 아니라 "이번 주에 무엇을 결정
해야 하는가"로 다시 정리할 수 있다.
세 번째 장면은 버그 수정이다. 재현 경로, 원인 가설, 수정 범위, 검증 계획이 작업 시작 메
모에 있고, 작업이 끝난 뒤 바뀐 점과 남은 위험이 인수인계 메모에 남아 있으면 Claude
Code는 단순 코드 생성기를 넘어서 검토와 이어받기 도구가 된다.
처음 시작할 때는 회의록 정리, 짧은 브리프, 작은 버그 수정처럼 원본이 분명하고 사람이
바로 다시 볼 수 있는 일부터 고르는 편이 안전하다. 고객 정보, 비밀값, 대외 발송, 실제 배
포처럼 한 번의 실수가 바로 영향으로 이어지는 일은 구조와 승인선을 익힌 뒤에 올리는 편

## p.7

1장. Claude 생태계가 왜 뜨거운가
1.1 왜 지금 다들 Codex, Claude Code, Gemini CLI를 같이 보는가
처음 막히는 지점은 이름이 너무 많다는 점이다. 왜 갑자기 Claude Code, Codex, Gemini
CLI 같은 이름이 한꺼번에 언급되는지, 왜 모델 점수보다 작업면과 자동화를 더 많이 이야
기하는지부터 헷갈리기 쉽다.
지금 이 시장은 모델 성능 하나로 설명하기 어렵다. 2026년 3월 23일 기준으로 Sonnet 4.6
과 Opus 4.6은 중요한 기준점이지만, 현장에서 체감하는 변화는 모델 숫자보다 AI가 어디
까지 실제 일을 이어서 할 수 있게 됐는지에 더 가깝다. 바이브 코딩(vibe coding) 이 자
주 언급되는 이유도 같다. 자연어로 코드를 만드는 데서 끝나지 않고, 테스트, 검토, 리팩터
링, 문서화, 업무 자동화까지 번졌기 때문이다.
그래서 지금은 OpenAI, Anthropic, Google을 함께 보게 된다. 경쟁 축이 "누가 더 똑똑한
가"에서 "누가 실제 파일과 테스트, Git, 브라우저, 외부 도구, 장기 세션을 더 잘 묶는가"로
옮겨갔기 때문이다. 모델이 더 똑똑해졌고, 더 오래 생각하는 추론과 확장 사고 계열이
붙었고, 파일·브라우저·외부 도구 연결이 더해지면서 답변이 실행으로 이어지기 시작했다.
Claude Code를 중심에 두는 이유도 여기에 있다. 파일, 규칙, 권한, 검증, 재사용, 원격 이어
받기, 외부 도구 연결, 팀 운영 같은 요소가 한꺼번에 모여 있고, Cowork까지 함께 보면 그
흐름이 코딩 밖 업무로 어떻게 넓어지는지도 같이 보인다.
연구 자료도 같은 결론을 낸다. METR의 Time Horizons와 HCAST는 에이전트가 처리할 수
있는 과업 길이가 빠르게 늘고 있지만, 긴 과업과 실제 맥락에서는 설정과 검증이 여전히 핵
심 병목이라고 말한다. Anthropic의 2026 Agentic Coding Trends Report도 비슷하다. 격차
는 모델 지식만이 아니라 누가 더 빨리 위임 규칙과 검증 하네스를 익혔는가에서 벌어진다.
같은 날 공개된 Anthropic의 Economic Index report: Learning curves도 이 흐름을 다른 각
도에서 확인해 준다. 2026년 2월 데이터에서는 Claude.ai 쪽 사용이 더 넓은 생활·개인 과업
으로 퍼지면서 상위 10개 작업의 집중도가 낮아졌고, 평균 과업 가치도 약간 내려갔다. 한마
디로 초기에는 코딩과 고부가가치 작업에 먼저 몰렸던 사용이, 이제는 더 넓은 층의 더 다양
한 일로 퍼지고 있다는 뜻이다. 반면 API 쪽은 코딩과 자동화가 더 강하게 이동했다. 즉 시
장은 동시에 두 방향으로 뜨거워지고 있다. 대중화는 더 넓게 퍼지고, 자동화는 더 깊게 파
고든다.

## p.8

여기서 말하는 하네스는 복잡한 이론이 아니다. 이 책에서는 먼저 Claude가 일하는 작업
장 정도로 보면 된다. 동료에게 일을 맡길 때 책상 위에 참고 자료를 놓고, 출입 가능한 서랍
을 정하고, 끝나기 전에 체크할 항목을 붙여 두는 것과 비슷하다. 어떤 파일을 읽게 할지, 어
디까지 수정할 수 있는지, 무엇을 확인한 뒤 끝낼지, 결과를 어디에 남길지를 묶어 둔 바깥
구조다. 모델이 강해질수록 "틀린 답"보다 "너무 넓게 고침", "검증 없이 끝냄", "건드리면
안 되는 파일까지 건드림"이 더 큰 문제가 된다. 그래서 실무에서는 모델 자체보다 작업 범
위와 검증선을 먼저 설계하게 된다.

"로그인 버그 고쳐 줘."

## p.11

기억과 대화 지속성이 강화되면서 Claude는 "한 번 질문하고 끝나는 도구"에서 "이어지는
작업을 다루는 도구" 쪽으로 이동했다. 그 다음에는 브라우저 확장, 파일 생성과 편집, 화면

운영 도구"다.

1.3 왜 지금 뜨거운가: 다섯 가지 구조적 이유
지금 시장은 "누가 더 말을 잘하느냐"보다 "누가 일을 덜 끊기게 하느냐"를 더 크게 본다. 아
래 다섯 가지 이유도 단순 기능 비교보다, Claude가 실제 업무 흐름 안에서 왜 많이 언급되
는지를 설명하는 기준에 가깝다.
이유 1. 채팅형 도구를 넘어서기 시작했다
ChatGPT류 제품에 익숙한 사용자는 AI를 질문-응답 박스로 보는 경우가 많다. 하지만
Cowork와 Claude Code는 질문에 답하는 화면보다, 일을 맡기고 결과 파일을 받는 화면으
로 이해하는 편이 실제 사용 방식과 더 가깝다.
이 변화는 사용자가 Claude를 다루는 기준도 바꾼다. 예전에는 "어떻게 물어볼까?"가 중심
이었다면, 지금은 "어떤 작업 공간과 규칙, 검증 루프를 만들까?"가 더 중요하다. 전자는 한
번의 답변 품질에 가깝고, 후자는 비슷한 품질의 결과를 반복해서 받는 문제에 가깝다.
이유 2. Skills와 Plugins가 재사용성을 높였다
Skill은 반복 절차를 담은 매뉴얼이고, Plugin은 그 절차와 실행 환경을 함께 배포하는 역할
묶음이다. 개인이 잘하던 방식을 팀이 같은 형태로 쓰기 시작할 때 이 차이가 커진다. 자세
한 구조 차이는 2장과 6장에서 다시 정리한다.
이유 3. MCP가 외부 세계와의 연결 비용을 낮췄다
Slack, Google Drive, Notion, GitHub 같은 도구가 MCP나 Connector로 붙으면 탭 전환이
줄고, 자연어와 실제 시스템 조작이 한 흐름으로 이어진다. 다만 연결이 늘수록 권한 표면도
넓어진다. 초반에는 "많이 붙인다"보다 "이번 주에 정말 자주 쓰는 연결 1~2개만 붙인다"
는 기준이 더 안전하다.
이유 4. 자동화 표면이 다양해졌다
같은 자동화처럼 보여도 시작 방식은 서로 다르다. Scheduled Tasks는 시간이 시작 조건이
고, Channels는 외부 이벤트가 시작 조건이며, Remote Control은 사람이 이미 돌아가는 세
션을 이어서 조작하는 기능에 가깝다. Claude Code on the Web은 새 세션을 원격으로 띄우
는 쪽으로 이해하는 편이 자연스럽다. 이 구분이 잡히면 어떤 일을 어느 표면에 배치해야 하
는지도 분명해진다.

이유 6. 이 경쟁은 기능 경쟁이 아니라 인프라 경쟁이기도 하다
지금 AI 시장을 제품 화면만으로 읽으면 중요한 절반을 놓치기 쉽다. 실제 경쟁은 모델 성능
과 UI만의 경쟁이 아니라, 누가 더 큰 데이터센터를 짓고, 더 많은 전력과 냉각 인프라를 확
보하고, 더 빠른 칩과 네트워크를 표준으로 만들고, 더 넓은 개발 생태계를 자기 스택 위에
올려 두는가의 경쟁이기도 하다. 미국이 GPU와 클라우드, 모델 생태계를 중심으로 표준을
선점하려 하고, 중국은 자립형 반도체와 모델 스택을 강화하며, 각국 정부와 기업은 데이터
센터와 전력 확보를 국가 전략처럼 다루기 시작했다. 이 흐름을 이해하면 왜 Claude 같은 제
품이 단순한 소프트웨어 기능 묶음이 아니라 더 큰 인프라 경쟁 위에 놓인 작업 표면인지 더
분명해진다.
이 관점은 실무자에게도 직접적인 의미가 있다. 예전에는 "어떤 SaaS를 도입할까"가 주된
질문이었다면, 이제는 "어떤 모델 제공자와 어떤 인프라 위에서, 어떤 규칙과 비용 구조로
계속 운영할까"가 더 중요한 질문이 된다. 같은 기능이 있어도 가격, 지연시간, 컨텍스트 길
이, 데이터 처리 정책, 기업용 관리 기능, 국가별 규제 대응이 다르면 실제 도입 판단도 달라
진다. 그래서 Claude 생태계를 읽을 때는 프롬프트나 UI만 보지 말고, 그 뒤에 있는 데이터
센터, 전력, 칩, 표준, 규제 경쟁까지 같이 보는 편이 정확하다.

Cowork
Cowork는 폴더와 파일 중심이다. 리서치, 문서 작성, 경쟁사 분석, 자료 정리, 보고서 초안,
파일 조직화에 적합하고, 비개발자도 채택하기 쉽다. AskUserQuestion, plugin, connector,
scheduled task 기반의 위임 흐름이 강해, 결과물이 파일로 남아야 하는 일에 특히 잘 맞는

다만 현재 제품 경계는 함께 알아두는 편이 좋다. 2026년 3월 기준 공식 도움말은 Cowork
를 유료 플랜 대상의 연구 미리보기로 설명하고, Claude Desktop for macOS와 Windows
x64를 기본 작업면으로 둔다.
둘은 상호 대체재가 아니라 서로 다른 작업층이다. 실제 고급 사용자는 둘을 함께 쓴다. 기
획안, 브리프, 리서치 초안은 Cowork에서 만들고, 구현, 통합, 테스트, 배포, 코드 리뷰는
Claude Code에서 처리한다. 중요한 구분은 난이도보다 위임 중심인지, 통제 중심인지다.

## p.29

1.6 "핫하다"는 말의 실체
기술 제품이 뜬다는 것은 단순 바이럴과는 다르다. 사용자 행동이 바뀌고, 운영 습관이 바뀌
고, 주변 도구 생태계가 붙고, 직무 분업이 재설계될 때 비로소 "핫하다"는 말이 실체를 갖
는다. Claude 생태계는 지금 이 네 조건을 거의 모두 충족하고 있다.

사용자 행동 변화
사용자는 프롬프트를 길게 쓰기보다 context file과 skill을 쌓기 시작했고, 하나의 세션만 쓰
지 않고 여러 세션과 여러 worktree를 병렬로 돌리기 시작했으며, 결과물만 받는 것이 아니
라 계획-실행-검증 루프를 설계하기 시작했다.

운영 습관 변화
CLAUDE.md 를 협업 규약처럼 유지하고, Hook으로 lint와 test, review를 강제하고, plan
파일과 handoff 파일, work log를 남기는 방식이 널리 보인다.

생태계 확장
Anthropic 공식 skills 저장소와 plugins 저장소, Agent Skills 오픈스탠다드, 각종 MCP 서버와
plugin marketplace, 서드파티 skill hub가 붙으며 모델 하나보다 주변 운영층이 커졌다.

직무 분업 재설계
마케팅, 세무, PM, 디자인, 보안, 연구, 콘텐츠 운영, 교육, 리서치 직무에서 Claude를 보조
챗봇이 아니라 자동화 계층으로 다루기 시작했다.
