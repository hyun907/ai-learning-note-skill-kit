## 1. 이 자료를 읽은 이유

클로드를 쓰면서 단순히 답변을 잘 받는 법이 아니라, 클로드를 실제 작업에 어떻게 안정적으로 연결해야 하는지 더 자세히 알고 싶었다.

최근 Claude Code와 Codex를 활용해서 학습 자료를 정리하고, 내가 이해한 내용을 README로 남기는 AI 학습 기록 워크플로우를 만들고 있다. 이 과정에서 AI에게 어디까지 맡기고, 어디서부터는 내가 직접 판단해야 하는지 구분하는 것이 중요하다고 느꼈다.

그래서 이 자료를 통해 Claude Chat, Projects, Cowork, Claude Code가 각각 어떤 역할을 하는지 이해하고, 내 학습 기록 워크플로우에 적용할 수 있는 구조를 찾고자 했다.

## 2. 내가 이해한 핵심

이 자료의 핵심은 Claude를 단순한 대화형 AI가 아니라, 실제 작업을 수행하는 시스템으로 봐야 한다는 점이다.

예전에는 “어떤 모델이 더 똑똑한가”가 중요했다면, 이제는 AI가 실제 파일, 테스트, Git, 브라우저, 외부 도구, 장기 세션과 얼마나 잘 연결되는지가 더 중요해졌다는 흐름이 인상적이었다. 즉, 좋은 결과는 모델 성능만으로 나오는 것이 아니라 작업 구조, 하네스, 검증 습관에서 나온다는 점을 이해했다.

특히 하네스 개념이 중요했다. AI에게 막연히 “이거 해줘”라고 요청하는 것이 아니라, 어떤 파일을 읽을지, 어디까지 수정할 수 있는지, 무엇을 확인하고 끝낼지, 결과를 어디에 남길지를 미리 정해줘야 한다. 결국 AI를 잘 쓰는 것은 프롬프트를 잘 쓰는 것뿐만 아니라, AI가 일할 수 있는 작업 환경과 기준을 설계하는 일에 가깝다고 느꼈다.

또 Claude를 사용하는 표면이 Chat, Projects, Cowork, Claude Code로 나뉜다는 점도 정리됐다. Chat은 빠른 질문과 답변에 적합하고, Projects는 같은 배경 자료를 계속 유지하는 데 적합하다. Cowork는 문서와 정리 결과물을 만드는 데 적합하고, Claude Code는 실제 코드 수정, 테스트, Git 작업처럼 개발자가 통제하며 진행해야 하는 일에 적합하다.

Cowork와 Claude Code의 차이도 단순히 쉬운 도구와 어려운 도구의 차이가 아니라, 위임 중심인지 통제 중심인지의 차이라는 점이 기억에 남았다.

## 3. 내 학습이나 프로젝트에 연결할 점

이 내용은 지금 만들고 있는 AI Learning Note Skill Kit과 직접 연결된다.

내 워크플로우에서도 AI가 모든 것을 대신 작성하게 하는 것이 아니라, AI가 정리할 영역과 내가 직접 써야 할 영역을 분리하는 것이 핵심이다. 예를 들어 `Learning Summary`는 자료에서 확인 가능한 내용을 AI가 정리하고, `My Note`는 내가 직접 이해한 내용과 프로젝트에 연결할 점을 적는 식으로 역할을 나눈다.

이 구조는 자료에서 말하는 하네스 개념과 비슷하다. AI에게 단순히 “이 자료 정리해줘”라고 하는 것이 아니라, 입력 파일, 출력 파일, 작성 범위, 금지사항, 검토 기준을 미리 정해두는 방식이기 때문이다.

앞으로 내 워크플로우를 더 안정적으로 만들려면 Claude Code를 Writer 역할로, Codex를 Reviewer 역할로 나누는 구조를 더 명확히 해야 한다. Claude Code는 자료를 읽고 Summary, My Note Guide, README Draft를 만드는 쪽에 두고, Codex는 AI가 내 해석을 지어내지 않았는지, 자료에 없는 내용을 단정하지 않았는지 검토하는 역할로 둘 수 있다.

또 `CLAUDE.md`, `plan.md`, `work log`, `handoff` 같은 파일을 활용하면 AI가 작업 맥락을 잃지 않고 이어갈 수 있을 것 같다. 지금은 워크플로우를 한 번 실행하는 데 집중하고 있지만, 이후에는 반복 가능한 작업 시스템으로 만드는 방향으로 확장하고 싶다.

## 4. 아직 헷갈리는 점

Cowork와 Claude Code의 차이는 개념적으로는 이해했지만, 실제로 어떤 작업을 Cowork에 맡기고 어떤 작업을 Claude Code에 맡겨야 하는지는 아직 완전히 감이 오지 않는다.

또 Skills, Plugins, MCP, Hooks, Agents 같은 개념들이 각각 어떤 역할을 하는지는 대략 알겠지만, 실제 프로젝트에서 언제 필요한지까지는 아직 구분이 어렵다. 특히 Skills와 Plugins의 차이, MCP와 Connectors의 차이, Hooks가 실제로 어떤 상황에서 유용한지 더 확인해보고 싶다.

하네스 개념도 중요하다는 것은 이해했지만, 내가 만든 학습 기록 워크플로우에서 하네스를 어느 정도까지 세부적으로 설계해야 하는지는 더 실험이 필요하다. 너무 느슨하면 AI가 임의로 판단할 수 있고, 너무 빡빡하면 매번 쓰기 불편한 워크플로우가 될 수 있을 것 같다.

## 5. 다음 액션

다음에는 지금 만든 AI 학습 기록 워크플로우를 하네스 관점에서 다시 정리해볼 것이다.

먼저 입력 파일, 출력 파일, AI가 작성해도 되는 영역, 사용자가 직접 작성해야 하는 영역, 검토 기준을 명확히 나누고 싶다. 특히 `Learning Summary`, `My Note`, `README Draft`, `Review Findings` 각각의 역할을 더 분명하게 정의할 것이다.

그리고 Claude Code에게 맡길 작업과 Codex에게 맡길 검토 작업을 나눠서 문서화할 것이다. Claude Code는 Writer로 두고, Codex는 Reviewer로 두는 구조를 README나 블로그 글에 설명할 수 있을 정도로 정리해보고 싶다.

마지막으로 다음 chunk를 읽을 때도 단순히 내용을 요약하는 데서 끝내지 않고, “이 개념을 내 워크플로우에 어떻게 적용할 수 있는가”를 계속 기준으로 삼아 기록할 것이다.
