## 1. 핵심 주장

- 이 책은 "AI 기능 백과사전이 아니라, 질문창에서 시작한 일을 실제 작업 시스템으로 옮기는 과정을 다룬다."(p.2)
- Claude 생태계의 경쟁 축이 "누가 더 똑똑한가"에서 "누가 실제 파일과 테스트, Git, 브라우저, 외부 도구, 장기 세션을 더 잘 묶는가"로 옮겨갔다고 설명한다.(p.7)
- "차이는 모델 이름보다 작업 구조, 하네스, 검증 습관에서 더 많이 벌어진다."(p.3)
- Claude를 "모델이 아니라 작업 시스템으로 다루기 시작했다"는 것이 1장의 결론이다.(p.30)

## 2. 중요한 변화

### AI 도구 경쟁 축의 이동 (p.3, p.7)
- "어느 모델이 더 똑똑한가" → "어느 도구가 실제 일을 덜 끊기게 하는가"로 중심 질문이 이동했다고 기술한다.
- 바이브 코딩(vibe coding)이 자연어 코드 생성에서 테스트·검토·리팩터링·문서화·업무 자동화까지 확장되었다고 설명한다.

### Claude의 확장 순서 (p.10–p.11)
자료는 2025년 하반기 이후 다음 순서로 Claude가 확장되었다고 정리한다.
1. 기억과 대화 지속성 강화
2. 브라우저 확장, 파일 생성·편집, 화면 이해 기능
3. 반복 작업을 묶는 기능 + 팀 배포 기능
4. 2026년 들어 Cowork(비개발 업무)와 Claude Code(개발 업무)가 별도 작업면으로 선명해짐
- 큰 흐름: "질문 도구 → 작업 도구 → 팀 운영 도구"

### 2025–2026 주요 업데이트 흐름 (p.11–p.13)
- 2025년 가을: 브라우저 이해, 파일 생성·편집, 멀티탭 작업
- 2025년 겨울: Skills·Agent Skills 공개 표준, Excel 연동, 긴 대화 압축, 예약 실행
- 2026년 1분기: Cowork 연구 미리보기, Claude Code Team 포함, 플러그인 장터, 관리자 제어, Enterprise 분석 기능, Excel/PowerPoint 추가 기능, Cowork 지속 세션

### Anthropic Economic Index 데이터 (p.7)
- 2026년 2월 기준, Claude.ai 사용이 더 넓은 생활·개인 과업으로 퍼지면서 상위 10개 작업의 집중도가 낮아지고 평균 과업 가치가 약간 내려갔다.
- API 쪽은 코딩과 자동화로 더 강하게 이동했다.
- 6개월 이상 사용한 고숙련 사용자 집단이 더 높은 가치의 과업과 업무 과업을 더 자주 가져오고, 같은 종류의 과업 안에서 대화 성공률이 더 높다.

## 3. 주요 개념

### 4개 작업 표면 (p.9–p.10)
자료는 Claude를 사용하는 표면을 다음 4가지로 구분한다.

| 표면 | 자료의 정의 |
|------|------------|
| **Chat** | 질문하고 바로 답을 받는 곳. "질문을 던지는 상담 창구"에 비유. |
| **Projects** | 같은 배경 자료를 계속 들고 가는 곳. "공용 배경 자료가 쌓인 캐비닛"에 비유. |
| **Cowork** | 자료를 읽혀 문서 초안이나 정리 결과를 받는 곳. "문서와 결과물이 오가는 실무 책상"에 비유. |
| **Claude Code** | 코드를 바꾸고 테스트하고 검토하는 곳. "공구와 검수표가 함께 놓인 개발 작업대"에 비유. |

### 하네스 (p.8)
자료에서 정의하는 하네스: "Claude가 일하는 작업장". 동료에게 일을 맡길 때 참고 자료를 놓고, 출입 가능한 서랍을 정하고, 체크할 항목을 붙여 두는 것에 비유. 구체적으로는 "어떤 파일을 읽게 할지, 어디까지 수정할 수 있는지, 무엇을 확인한 뒤 끝낼지, 결과를 어디에 남길지를 묶어 둔 바깥 구조"로 설명한다.

- 하네스 없는 요청 예: "로그인 버그 고쳐 줘."
- 하네스 있는 요청 예: "`auth/` 폴더만 읽고, `.env`는 건드리지 말고, 로그인 테스트가 통과하면 변경 이유를 세 줄로 남겨라."

### 제품별 정의 (p.21–p.23)

| 제품 | 자료의 정의 |
|------|------------|
| **Claude Chat** | 기본 대화 경험. 아이디어 발상, 요약, 초안 작성, 학습, 빠른 분석에 적합. |
| **Projects** | 특정 프로젝트의 파일과 문맥을 묶는 공간. 전제와 참고 자료를 오래 유지. |
| **Cowork** | 지식업무 중심의 로컬 작업 자동화 층. 폴더와 파일을 바탕으로 여러 단계를 이어 수행. 문서·리서치·정리·보고서·자료 생성·반복 업무에 설계. |
| **Claude Code** | 터미널 기반의 개발 에이전트. 파일·쉘·Git·MCP·hooks·plugins·subagents·agent teams 조합. |
| **Skills** | Claude가 특정 작업을 일관되고 빠르게 수행하게 만드는 재사용 가능 작업 흐름. "이 일은 이런 순서와 이런 기준으로 처리한다"는 업무 매뉴얼 폴더에 비유. |
| **Plugins** | Skills, commands, agents, hooks, MCP, LSP 등을 묶는 배포 단위. Skill이 절차 파일 묶음이라면, Plugin은 절차와 실행 환경을 함께 배포하는 단위. |
| **MCP / Connectors** | 외부 앱과 데이터를 Claude에 연결하는 인터페이스. "Claude가 어디에 접근 가능한가"를 결정. |
| **Hooks / Agents / Agent Teams** | Claude가 "어떻게 통제되고 검증되는가"를 결정하는 층위. Hooks는 클라이언트 수준에서 행동을 고정. |
| **Remote Control** | 사람이 멀리서 이미 돌아가는 세션을 이어받는 방식. |
| **Channels** | 외부 시스템이 사건을 밀어 넣는 방식으로 작업을 시작. |
| **Scheduled Tasks** | 시간이 자동으로 일을 시작시키는 방식. |

### Cowork와 Claude Code의 차이 (p.27–p.28)
자료는 "Cowork는 쉬운 버전, Claude Code는 어려운 버전"이 아니라 "작업 모델이 다르다"고 설명한다.
- Claude Code: 터미널 중심. 개발자가 통제권을 강하게 쥠. 사람이 중간 단계를 계속 살피며 고쳐야 하는 일에 적합.
- Cowork: 폴더·파일 중심. 비개발자도 채택 쉬움. 결과물이 파일로 남아야 하는 일에 적합. 2026년 3월 기준 유료 플랜 대상 연구 미리보기, Claude Desktop for macOS·Windows x64 기본 작업면.
- 구분 기준: "난이도보다 위임 중심인지, 통제 중심인지"

### Computer Use 공식 상태 (p.11)
- 자료 기준(2026년 3월 24일): 일반 claude.ai 채팅창이 아닌 상용 API의 베타 실행 도구.
- `computer-use-2025-11-24`를 Opus 4.6, Sonnet 4.6, Opus 4.5용 기준선으로 사용.
- 전용 샌드박스와 사람 감독을 기본 전제로 설명.
- Claude in Chrome(브라우저 보조), Cowork(데스크톱 작업면), Computer Use(API 화면 행동 위임)는 서로 다른 층.

### OpenClaw 비교 (p.13–p.14)
자료는 OpenClaw를 "경쟁 제품"이 아니라 "사용자가 이미 어떤 작업 방식을 원하고 있었는지 먼저 보여 준 사례"로 제시한다. Anthropic은 OpenClaw가 드러낸 수요를 자사 제품 철학에 맞게 여러 표면으로 나눠 제품화하고 있다고 설명한다.

| OpenClaw | Anthropic 대응 |
|----------|----------------|
| Gateway + multi-channel inbox | Channels, Remote Control, Claude Code on the web, Cowork persistent thread |
| cron jobs + wakeups + heartbeat | Claude Code scheduled tasks, Cowork scheduled tasks, Channels |
| workspace/local/내장 skills | skills, plugins, organization-managed plugin, marketplace |
| 원격 gateway / SSH / ws | Remote Control(로컬 세션 이어받기), Claude Code on the web(원격 VM 작업 위임) |

## 4. 적용 아이디어 후보

자료에 명시된 적용 사례와 예시만 옮긴다.

### 직무별 첫 적용 지점 (p.24)

| 직무 | Chat | Projects | Cowork | Claude Code |
|------|------|----------|--------|-------------|
| 창업가/1인 사업가 | 생각 정리 | 배경 지식 누적 | 시장 브리프와 초안 | MVP 수정과 검증 |
| 비개발 실무자 | 빠른 요약 | 반복 배경 유지 | 회의록, 브리프, 정리 자동화 | 거의 나중 단계 |
| PM/기획 | 질문 정리 | 기능 배경 유지 | PRD 초안, 회의 정리 | 개발팀과 같이 볼 때만 |
| 개발자 | 빠른 아이디어와 설명 | 저장소 배경 묶음 | 보조적 | 버그 수정, 테스트, 리팩터링 |
| 연구자/분석가 | 개념 설명 | 논문 배경 누적 | 브리프와 정리 | 재현 환경, 실험 하네스 |

### 좋은 첫 과제 예시 (p.25)
- Claude Chat: "이 보고서 초안을 5문장 요약과 3개 질문으로 정리해 줘"
- Projects: "이 프로젝트 폴더를 읽고, 앞으로 계속 참고할 핵심 배경지식 한 페이지를 만들어 줘"
- Cowork: "이 폴더의 회의 메모를 읽고 주간 브리프 초안을 `outputs/`에 저장해 줘"
- Claude Code: "이 버그 재현 경로를 읽고, 원인 가설과 수정 계획을 먼저 `plan.md`로 정리해 줘"

### 나쁜 첫 과제 예시 (p.25)
- 처음부터 모든 connector, plugin, MCP를 한꺼번에 켜기
- 템플릿 없이 "우리 회사 톤으로 다 알아서 해 줘" 말하기
- 코드베이스를 읽히자마자 신규 기능 전체 구현 맡기기
- 사람 검토 경계 없이 메일 발송·삭제·정산 입력까지 바로 자동화하기
